The problem

  • OME project built monolithically
  • Project components
    • either embedded in the same source code
    • or managed as submodules
  • Version management
  • Inability to releases
  • Multiplication of epics/spaces

The problem

Example 1: Bio-Formats

  • Bug reported for a file format
  • PR opened, tested and merged in the development line
  • Bio-Formats release on-hold until OMERO can be released

The problem

Example 2: OMERO

  • Security vulnerability
  • PR opened, tested and merged in the development line
  • OMERO released
  • Bio-Formats version number

General considerations

To achieve full decoupling, the following components need to be decoupled

  • Source code repository
  • Build system
  • Continuous integration
  • Release

Bio-Formats decoupling

Source code repository

  • Bio-Formats has its own source code repository
  • Bio-Formats build system/release process are completely autonomous

Bio-Formats decoupling

Build system

  • Submodule introduced in November 2011
  • Bio-Formats source code embedded into OMERO source code
  • Reference to the revision via the SHA1
  • Bio-Formats is built every time
  • Version management previously defined via ivy.xml
    ivy.xml<publications defaultconf="build,runtime">
    <artifact name="formats-gpl"/>
...
<dependencies defaultconfmapping="build,runtime->default">
    <dependency org="bf-deps" name="xalan" rev="2.7.1"/>

Bio-Formats decoupling

Build system

  • See Bio-Formats build decoupling PR opened
  • Submodule removed in June 2015
  • Bio-Formats artifacts released to the OME downloads and the OME artifactory via Maven
  • Bio-Formats version managed via Ivy/Maven

Bio-Formats decoupling

Build system

  • Bio-Formats artifacts released to the OME downloads and the OME artifactory via Maven
  • Version declared in 
    omero.propertiesversions.bioformats=5.1.2
  • Individual Ivy files declare dependencies 
    ivy.xml<dependency org="bf-deps" name="xalan" rev="2.7.1"/>
